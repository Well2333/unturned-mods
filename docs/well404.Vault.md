# well404.Vault

Vault 的玩家菜单和管理员容量管理采用插件自带 HTML/CSS/JavaScript，由 WebPanel 在 Shadow DOM
中挂载。物品目录在常规桌面宽度为六列，并随窗口逐级降列，仍复用既有存取 handler、权限检查与 SQLite 数据。

提供个人仓库和小队共享仓库。物品**完整保真**保存（耐久、品质、枪械配件、弹匣/弹药箱内的弹药等），
取出原样还原；容量按**背包格子数**计。

## 依赖

- 个人仓库无硬依赖，可独立运行。
- **可选**：[`well404.Essentials`](well404.Essentials.md) → 提供实时小队身份，小队共享仓库才会出现。
- **可选**：[`well404.Economy`](well404.Economy.md) → 允许在线玩家用 SQLite 余额或游戏经验值购买个人/小队容量。
- **可选**:[`well404.WebPanel`](well404.WebPanel.md) → 提供玩家网页仓库与容量的可视化设置。

```bash
openmod install well404.Vault
```

安装或升级 DLL 后必须完整重启服务器；`openmod reload` 只适合未替换二进制时的配置重载。

## 命令

> OpenMod 按命令路径派生权限,需在权限角色里授予。

| 命令 | 语法 | 权限 | 说明 |
| --- | --- | --- | --- |
| `/vault`（`/storage`） | | `well404.Vault:commands.vault` | 显示用法 |
| `/vault store`（`deposit`） | `<物品ID> [数量]` | `well404.Vault:commands.vault.store` | 把背包里该物品存入仓库 |
| `/vault take`（`withdraw`） | `<物品ID> [数量]` | `well404.Vault:commands.vault.take` | 把仓库里该物品取回背包 |
| `/vault list`（`info`） | | `well404.Vault:commands.vault.list` | 列出仓库内容与占用 |
| `/vault team store`（`deposit`） | `<物品ID> [数量]` | `well404.Vault:commands.vault.team.store` | 存入当前小队共享仓库 |
| `/vault team take`（`withdraw`） | `<物品ID> [数量]` | `well404.Vault:commands.vault.team.take` | 从当前小队共享仓库取出 |
| `/vault team list`（`info`） | | `well404.Vault:commands.vault.team.list` | 列出当前小队仓库 |
| `/vault team upgrade`（`buy`） | | `well404.Vault:commands.vault.team.upgrade` | 用自己的余额购买小队容量 |

- **按游戏物品 ID 操作**:`/vault store 363 2` 存入 2 个物品 363;`/vault take 363` 取回 1 个。
- **只存背包**:只会存入**背包 / 穿戴容器**(背包、背心、衣裤等)里的物品;**手持的主/副武器槽物品不会被存入**,需先放进背包。
- 取出时若背包放不下,物品会**掉落在玩家脚下**(不会丢失)。
- `store` 一次最多存入背包中匹配该 ID 的若干个;若仓库容量不足会**存到放不下为止**并提示已满。
- **网页与命令权限一致**:网页仓库的存 / 取分别需要 `commands.vault.store` / `commands.vault.take` 权限(与命令相同);
  缺少权限时网页仓库**只读**(可查看,按钮不显示),无法借网页绕过命令权限。
- 小队网页操作严格使用 `.team.store` / `.team.take` / `.team.upgrade`，不会复用个人仓库权限。

## 小队共享仓库

- 玩家当前只能属于一个 Unturned 小队；Vault 每次操作都从服务器实时解析当前小队，不信任浏览器缓存或客户端提交的小队 ID。
- 仓库主键由提供方、服务器作用域和稳定 `groupID` 组成，小队改名不改变仓库归属。
- 默认基础容量 200 格、最大 5000 格。任何有升级权限的成员都可用自己的余额购买容量；购买所得归小队，离队不退款。
- SQLite 容量预留与物品写入在同一事务内，多成员同时存入不会突破容量上限。
- 游戏背包与 SQLite 无法组成同一个 ACID 事务，因此每次存取都会写入 `vault_transfer_audit`，并在
  `vault_transfer_items` 中保存每件物品的 ID、内部数量、品质、原始状态、格子占用和最大内部数量。正常操作标记完成，
  明确未提交的失败会返还物品并标记补偿；若调用方遇到模糊异常，会先查询同一事务写入的审计状态，只有确认数据库未提交时才补偿。
  进程在游戏背包与数据库之间崩溃时保留完整未完成记录并在启动日志告警，绝不自动重放不确定状态，避免复制物品。
  已完成/已补偿记录保留 30 天后自动清理。
- 新购买在同一 SQLite 事务内直接进入 `debiting`，再用 Economy 的唯一操作号幂等扣款，
  持久化“已扣款”后重新确认买家仍属于原小队，才进入可扩容的 `ready`。实例内恢复与在线购买共享串行门，
  数据库 CAS 则负责 reload 新旧实例短暂并存时的正确性。离线买家的记录保持待恢复；确认玩家在线但无队伍或已换队时自动退款。
  Vault 会在依赖插件加载后、每 30 秒以及买家下次购买前继续恢复旧记录。周期恢复每批最多处理 25 条，购买前只检查该买家的前 5 条；Economy 与小队身份 provider 的单次恢复调用最多等待 3 秒，避免外部依赖卡住全局购买门。
  若扣款等待期间玩家离开或切换小队，CAS 会把操作改成 `refund_pending`，只有成功取得退款权的一方才能退款；
  已 `completed` 的操作不能再进入退款状态。
- Economy 缺失时付费扩容不可用；`database` 与 `experience` 后端均可购买。经验值模式只支持在线玩家且没有跨重启幂等账本，极端中断记录会留在恢复隔离区供人工核对；经验值没有小数，因此扩容价格必须是整数。`price: 0` 的免费扩容不依赖 Economy。
- Essentials 暂时卸载或 reload 时，小队操作安全返回不可用，个人仓库继续工作；插件二进制升级仍必须完整重启。

## 容量与计费(按格子)

- 每个存入的物品占用其**网格尺寸**(`size_x × size_y`):例如 1×1 的小物品占 1 格、2×2 的弹药箱占 4 格。
- **物品内部的堆叠数 / 弹药数不计入占用**:一盒装着 39 发子弹的弹药箱,仍只按它的格子尺寸占用,
  **不会**被算成 39 个物品。读不到某物品尺寸时,该物品按 1 格计。

### 每位玩家的容量（基础、权限档位、购买与管理员调整）

个人仓库默认基础容量为 50 格。玩家取配置基础容量与所持权限档位中的最大值，再加上已购买格数；默认每次花费 100 购买 10 格，最高 5000 格。管理员不再维护独立的“玩家覆盖”目录，而是在“仓库查看”中直接编辑当前个人或小队仓库的有效容量。该调整以容器增量保存，不会清除既有购买记录，之后继续购买仍会增加容量。 所有基础容量、档位、购买增量和管理员设置均受 1–1,000,000 格硬上限约束，计算使用 64 位中间值，数据库读取和运行时写入都会拒绝越界值。升级到数据库 schema v6 时，旧 `capacity_overrides(steam_id, capacity)` 会迁移为兼容的绝对容量（包括没有任何库存行的玩家）；后续基础/权限档变化不会改变这个旧绝对值，而迁移后的新购买仍继续累加。迁移在同一事务内推进版本且可重复执行；已在 v5 中购买或编辑过容量的容器不会被遗留表覆盖。

## 完整保真

存入时记录物品的 **id、堆叠数(amount)、品质(quality)、原始状态字节(state)**;取出时用这些原样重建,
因此**枪械的配件、弹匣里的弹药、弹药箱内的余弹、物品耐久**等都会保留。

每个 Unturned 物品实例都携带原始 `quality` 字节，但它不一定代表耐久。网页只在物品资源的
`ItemAsset.showQuality` 为 `true` 时显示耐久并按其区分变体；夹板、医疗品、补给箱等没有耐久语义的
物品会忽略该字节进行界面合并，同时数据库仍原样保存它，取出时不会丢失任何原始数据。仓库数据以规范化表持久化到插件数据目录下的 `vault.sqlite3`，物品原始状态以 SQLite BLOB 保存。
2.0.0 不读取或迁移旧 `vault.data.yaml`。

部分旧创意工坊包会错误地让多个不同资源共用同一个 16 位物品 ID。仓库记录与游戏物品实例本身只能
保存这个数字 ID，无法额外保存冲突资源的 GUID；因此名称、中文翻译、稀有度、类型、分类和
`showQuality` 必须与游戏对该 ID 的实际解析保持一致。Vault 使用
`Assets.find(EAssetType.ITEM, id)` 选中的 `ItemAsset` 作为唯一权威来源，不依赖 OpenMod 目录枚举
重复资源时的不稳定先后顺序。

## 配置（`config.yaml`）

```yaml
# 基础容量(背包格子数);每个物品按其网格尺寸(宽×高)占用;内部堆叠/弹药数不计入。
maxSlots: 50

# 按权限的容量档位;玩家取「基础容量与其拥有的各档位」中的最大值。空 = 无档位。
tiers: {}
# 示例：
# tiers:
#   "well404.vault.size.vip": 400
#   "well404.vault.size.mvp": 600

personalPurchase:
  enabled: true
  maxSlots: 5000
  slotsPerPurchase: 10
  price: 100

teamVault:
  enabled: true
  baseSlots: 200
  maxSlots: 5000
  purchase:
    enabled: true
    slotsPerPurchase: 10
    price: 500
```

装了 WebPanel 时，个人/小队基础容量、购买开关、购买格数、价格和上限均可编辑；当前仓库的有效容量在“仓库查看”中调整。

## Web 面板(可选)

装了 [`well404.WebPanel`](well404.WebPanel.md) 后:

- **管理面板**:注册「仓库」模块(图标 🧳):
  - **设置**:个人基础容量 `maxSlots`、权限档位 `tiers`，以及小队开关、基础/最大容量、每次购买格数和价格;
  - **小队仓库**:只读查看稳定小队键、显示名、当前占用、总容量、已购买容量、状态、容器版本和待恢复购买数；
    超容量或存在待恢复购买时会显示醒目标记。
  - **小队仓库（含恢复隔离区）**：小队概览下展示每笔待恢复购买的操作号、买家、小队、价格、格数、持久阶段和时间。管理员可对可证明未扣款的持久化购买执行“安全中止”（若另一实例仍在扣款，会用同一幂等操作同步后退款），对 `refund_pending` 执行幂等“重试退款”，或在已线下核实退款后执行“确认已退款”。每次操作必须填写审计原因并再次完整输入操作号；服务端使用预期状态 CAS，状态转换和 `team_vault_purchase_resolutions` 追加审计在同一 SQLite 事务内完成。经验值后端没有持久账本，禁止自动判定未扣款或自动重试；管理员核对并人工退款后，可对 `refund_pending` 或经验值 `debiting` 记录执行确认。页面同时展示中断存取的
    操作者、方向、容器、每件物品的完整载荷和逐项处理阶段。`removal_started` / `delivery_started` 表示进程可能恰好在
    Unity 背包变更与 SQLite 记账之间中断，必须人工核对；所有状态不确定的存取禁止自动重放。
- **玩家网页仓库**:注册一个面向玩家的「仓库」菜单(玩家面 `/p`,图标 🧳)。玩家 `/menu vault` 收到链接,
  在浏览器里看到:
  - 个人/小队仓库标签旁显示当前「X / Y 格」，购买容量按钮紧邻容量信息；页面顶部不再重复占用一行显示容量;
  - **背包**分区(存入)与**仓库**分区(取出),每个物品一行;
  - **相同的合并**:同种物品中**属性完全一致**(相同弹药数 / 耐久 / 配件)的若干件**合并为一行**(显示 `×件数`),
    不再逐件铺开,也没有展开按钮;
  - **不同属性默认收起**:同种物品若存在**不同属性**的多个变体(如 5 发和 8 发的弹药盒),则折叠成一张汇总卡片,
    点击汇总卡后在**弹窗**中展示各变体；每张变体卡只显示其区别属性——弹药/堆叠的
    **装填比例**(如 `6/8`,避免与件数混淆;读不到容量时退回原始数量)、耐久(`耐久 X%`)、`🔧`(带枪械配件等
    内部配置)+ 该变体的件数(`×N`);
  - **存取操作**:某条目只有 **1 件**时只给一个「**存入/取出**」按钮(没有可选数量,不再有重复按钮);有**多件**时
    给「**全部 / 数量 / 一件**」——全部=该(变体/物品)全存或全取;数量=弹出一个**空白**输入框(标题标注可填范围
    `1–N`),必须自行输入个数,因此**与「全部」「一件」都不同**;一件=只操作一件。变体行只作用于**该变体**
    (于是 5 发盒与 8 发盒可分别批量存取)。在变体弹窗中点击「数量」会再打开上层数量输入窗口。
  - **稀有度**:卡片边框使用物品资源的 Unturned 原生稀有度颜色（普通、罕见、稀有、史诗、传说、神话），
    卡片上同时显示本地化稀有度文字，因此不依赖颜色作为唯一辨识方式；变体弹窗内沿用同一标识。
  - **排序**:可按「总占用格数、物品 ID、物品数量、物品稀有度、名称」排序并切换升/降序。
    默认按**总占用格数降序**，相同值以物品 ID 稳定排序；选项保存在当前浏览器会话中。
  - **两级分类筛选**:第一行保留「弹药、食物与饮品、药品、武器、材料、工具、服装、配件、建筑、
    载具用品、其他」大类，第二行按物品资源的原生 `ItemAsset.type` 显示准确小类；选择大类后只显示
    该范围内当前存在的原生类型。分类不再读取配方操作，不使用名称或 ID 猜测；`Supply` 始终按自身
    原生类型进入材料。未知新类型显示原始类型文本并进入「其他」，不会从列表消失。两行标签都显示
    当前背包或仓库内的实际件数，没有物品的类别不会占用界面空间。
  - **身上容器筛选**:背包分区提供手部、背包、胸挂、上衣、裤子五个可独立开关的来源按钮，默认全部开启。
    例如关闭「裤子」与「手部」后，只展示并存入背包、胸挂和上衣中的物品；同 ID、同状态物品位于不同容器时也不会
    混成同一张卡，卡片会标明来源，存入动作只匹配该来源页，不会误取被隐藏容器里的同类物品。
  - **状态记忆**:当前个人/小队与背包/仓库标签、排序字段、排序方向、大类/原生类型筛选和容器开关在当前浏览器会话内保留；
    切换分区时若所选类别不存在，会自动回到「全部」。
  - **布局**:背包/仓库使用顶部标签切换；常规桌面宽度使用六列紧凑等高卡片，并随窗口降为 5/4/3/2/1 列。长名称会在卡片内换行，同一行卡片仍保持等高。
  - 存取须玩家在线(操作其实时背包)且**持有对应权限**；个人与小队的 store/take/upgrade 权限完全独立;
    **仅查看**仓库内容则离线 / 无权限也可(此时按钮不显示)。该页面完全由通用的玩家菜单模型驱动
    (分区/列表/徽章/按钮/父子卡片数据),WebPanel 宿主不含仓库专属逻辑。

## 管理员仓库查看与编辑

1. 打开管理员面板的「仓库查看」。页面上方可在个人仓库与小队仓库之间切换，并列出所有当前有库存的玩家或小队；玩家条目会显示服务器最后记录到的游戏内角色名与 Steam 名。点击快捷卡片即可载入。
2. 仍可手工输入任意 17 位 SteamID 或稳定小队 ID，因此即使目标当前没有库存、未出现在快捷列表中，也能直接查看和编辑。
3. 当前仓库加载后，可把有效容量修改为不小于已占用格数、且不超过 1,000,000 的正整数。容量修改保存为容器调整量，不会清空已经购买的格数。
4. 点击「添加物品」可按物品 ID、物品内部数量、品质和份数创建完整的默认游戏状态记录；物品卡片支持修改、逐条删除，批量栏可按物品 ID 删除当前仓库的全部匹配记录。删除操作都有二次确认。
5. 管理页使用与玩家面板一致的胶囊标签、全宽响应式卡片和弹窗交互，弹窗只有在遮罩外按下并在遮罩外释放时才关闭。「恢复隔离区」直接位于「小队仓库」标签内，不再单独占用导航项。

所有写操作都由服务端按仓库类型和所有者键限定；单条操作还会校验数据库记录 ID。批量删除只影响当前仓库及指定物品 ID，不会扫描或修改其他仓库。玩家昵称保存在同一 `vault.sqlite3` 的 `vault_owners` 表中，仅作为管理员快捷检索元数据，不影响仓库物品。
