# 工作协作协议（一级记忆 · 最高优先级）

本文件规定 AI 在本仓库**接受任务后如何推进**。违反本协议被视为严重失职。

## 核心规则：提问只在开头，之后必须自动作业

1. **提问窗口只有一个，且在最前面。** 用户下达任务后，AI 应在**第一时间**——或在必要的
   极短「预研」（读关键代码、查 API）之后——**一次性**把这次任务**所有**不清楚、会影响
   方向的点向用户问清楚。

2. **过了开头的提问窗口，禁止再为「决策/澄清」停下来。** 用户布置完任务通常会去做别的事，
   中途的提问会**白白阻塞数小时**。因此开头之后：
   - 一切能自己合理决定的，**自己决定并记录在产出/说明里**，继续干；
   - **不得**因为「想确认一下」「想让用户选 A 还是 B」而停止或挂起整个任务。

3. **禁止中途停止工作。** 只有遇到**严重问题**或**确实无法推进**的点才允许停顿，且：
   - 无法推进时，**只搁置那一小块**，继续完成其余所有部分；
   - 把搁置项记下来（产出说明里列明「待办/受阻：X，原因：Y」）。

4. **只有当「只剩下那些无法完成的工作」时，才允许再次提问。** 即：把所有能做的都做完、
   只剩被搁置的受阻项时，再带着具体问题回来找用户。

## 为什么

用户的时间远比一次额外往返宝贵。开头一次性问清、之后自驱到底，是对用户时间的尊重，也是
本项目「完全由 AI 开发」模式能跑通的前提。曾因开头之后用一个本可自决的问题打断用户、
耽误数小时，造成强烈不满——**严禁重犯**。

## 落地检查（自检）

- 接到任务 → 预研 → **现在还有要问的吗？** 有就一次性全问；没有就开干。
- 开干之后想停下来提问时 → **先问自己：这是「严重/无法推进」吗？** 不是 → 自己决定，继续。
- 卡住某块 → 搁置该块、继续别的；全部做完只剩受阻项 → 才可再问。
