# 确立「分支化开发 + squash 合并」工作流

## 做了什么
- 在 `memory/guidelines/commit-and-memory-workflow.md` 新增「分支工作流与 squash 合并（强制）」一节：
  - 每项工作（新功能/新插件/bug 修复/文档/规范）都先开新分支（`feat/`、`fix/`、`chore/`、`docs/`），
    不在 `main` 上直接开发；
  - 尽量不直接 commit 到 `main`，除非极小且必要、不值得单开分支；
  - 合并回 `main` **一律用 squash merge**（一分支=一提交），避免碎片/试错提交污染主线；
  - changelog 跟随 squash 提交：两步法中的「功能提交」= squash 合并提交，变更记录用其短哈希命名；
  - 合并后删除分支（本地 + 远程）；推送 `main`/删远程分支同受「最高红线」需用户授权。
- 同步在「标准流程（两步提交）」段落加了交叉说明，指明 squash 时「功能提交」即 squash 合并提交。

## 为什么
- 用户要求：每次开新功能/新插件/bug 修复都开新分支，尽量不直接 commit，合并用 squash 避免
  commit 污染主线历史。

## 影响的文件 / 范围
