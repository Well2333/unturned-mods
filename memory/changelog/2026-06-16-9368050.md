# WebPanel:cloudflared 下载加连通性探测 + jsDelivr 镜像 + 按稳定性排序的多源

## 做了什么(well404.WebPanel 1.2.0 → 1.2.1)
- **连通性探测优先**:正式下载前对所有源做**并发探测**(1 字节、8s 短超时的 ranged GET),
  **只对可连通的源按序下载**(每源重试 `downloadAttempts` 次);可连通的全失败或一个都不通时,
  才把**未连通的源当兜底**直接再试。避免在被墙/失效源上耗满 60s 下载超时+重试。
- **jsDelivr CDN 镜像**:新增 jsDelivr 源,指向自建镜像仓库 `Well2333/asset-mirror`——其每日
  GitHub Action 检测 cloudflared 新版,拉取各平台二进制并 `gzip` 压缩后提交(Windows 版从 51.66MB
  压到 ~18MB,稳进 jsDelivr 50MB 单文件上限),再由 jsDelivr 全球 CDN 分发。插件对 URL 以 `.gz`
  结尾的源**下载后自动 gunzip**(`GZipStream`,netstandard2.1 内置,无新依赖)。
- **默认源顺序**:GitHub 本体 → jsDelivr 镜像 → 代理(`gh-proxy.com`、`gh.ddlc.top`);实测剔除了
  失效/不稳的代理(`ghproxy.net` 502、`mirror.ghproxy.com` 等无法连、`ghproxy.com` 跳落地页、
  `gh.con.sh` 停用),按实测稳定性排序。
- **修复**:`downloadMirrors` 里的空串 `""`(直连 GitHub 的标记)此前被 `!IsNullOrWhiteSpace`
  过滤掉,导致用户无法在自定义列表里用 `""` 固定 GitHub 优先;改为只跳过 `null`、保留 `""`。

## 为什么
- 真机被墙时直连 github.com 连接超时 ~21s;探测先行可快速跳过死源,直接走能连通的镜像。
- jsDelivr 是稳定的全球 CDN,但只服务 git 树文件、不服务 release 资产且单文件 ≤50MB;用「gzip 进
  自建镜像仓库」绕开这两条限制,得到一个被墙网络下也稳的来源(用户需求)。

## 影响的文件 / 范围
- `well404.WebPanel/CloudflaredDownloader.cs`(探测 ProbeReachableAsync、两段式 TryDownloadAnyAsync、
  CreateHandler、gunzip、默认源、空串修复)
