# WebPanel:修复 cloudflared 已缓存时隧道永不就绪(异步上下文死锁)→ /menu 一直「无公网地址」

## 做了什么(well404.WebPanel 1.2.3 → 1.2.4)
- **根治隧道卡死**:`OnLoadAsync` 改为用 `Task.Run` 在线程池启动整条隧道后台链路(初始
  SynchronizationContext 为 null),并补上 `await provider.StartAsync(...)` 等遗漏的
  `ConfigureAwait(false)`。
- **/menu 短暂等待**:隧道刚启动的窗口内,`/menu` 会等待隧道就绪(最多 ~25s)再生成链接,
  而不是立刻报「无公网地址」。实现为具体类 `PlayerWebSessionManager.CreateLinkAsync` +
  `BeginTunnel`/`SetTunnelUnavailable`;`CommandMenu` 注入**具体类型**(不动共享接口)。

## 为什么
- 真机日志(1.2.3)显示:`using the portable cloudflared` 之后**再无任何隧道日志**,12 分钟后
  `/menu` 仍无地址 —— 隧道流程卡死。根因:cloudflared **已缓存**时 `EnsureAsync` **同步返回**
  (未真正 await),后续 `await provider.StartAsync(...)`(漏了 `ConfigureAwait(false)`)的续体被
  post 回 **UniTask 主线程上下文**,而它不调度普通 `Task` 续体 → 永久卡住 → `SetTunnelBaseUrl`
  从不调用 → `/menu` 永远无地址。
- **为何之前没复现**:本地测试每次删缓存强制下载,走「下载」路径(带 `ConfigureAwait(false)`,
  脱离主线程)不卡;真机第二次开服缓存命中,走「同步返回」路径才卡。第一次能用、第二次起不来
  完全吻合。
- 共享接口不加方法:`UnturnedMods.Shared.dll` 被每个插件各自打包,改接口后只升级 WebPanel 可能
  在运行时加载到其它插件带的旧 dll(无新方法)→ `MissingMethodException`。故 `CreateLinkAsync`
  只放在 WebPanel 内部具体类上。

## 影响的文件 / 范围
- `well404.WebPanel/WebPanelPlugin.cs`(Task.Run 启动隧道 + ConfigureAwait(false) 补全 +
  BeginTunnel/SetTunnelUnavailable 调用)
- `well404.WebPanel/PlayerWebSessionManager.cs`(`m_TunnelReady`、`BeginTunnel`、
  `SetTunnelUnavailable`、`CreateLinkAsync`、`BuildLink`)
- `well404.WebPanel/Commands/CommandMenu.cs`(注入具体类型,`await CreateLinkAsync`)
- `well404.WebPanel/docs`、`.csproj`(1.2.4)
- 共享库 `IPlayerWebSessionService` **未改动**(撤销了之前的接口添加)。

## 注意事项 / 后续
- 实测验证:**保留 cloudflared 缓存**开服(复现真机第二次开服场景),隧道约 5 秒 `tunnel up`,
  不再卡死;整解 `-c Release` 0 警告 0 错误。
- 仅 WebPanel 改动,本轮只发布 well404.WebPanel 1.2.4;其余插件不变,无 shared dll 兼容问题。
- 旧的 `menu:unavailable` 英文文案仍可能停留在用户磁盘的 `translations.yaml`(OpenMod 首次落盘后
  不覆盖);修复后大多数情况不会再触发它。
