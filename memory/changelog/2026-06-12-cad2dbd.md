# Essentials 新增 /party 指令组队（邀请 + 接受）

## 做了什么

- 新增 `/party` 一组指令，用聊天指令直接组队，绕过 Unturned 难用的游戏内组队菜单：
  - `/party`（看在线队员与角色）、`/party invite <玩家>`、`/party accept [玩家]`、
    `/party deny [玩家]`、`/party leave`、`/party kick <玩家>`（仅队长 ADMIN）。
- 加入模式为**邀请 + 接受**（用户选定，防滥用）：`PartyInviteManager` 复用 tpa 那套
  「按收件人存挂起请求 + 超时通知 + 实例精确移除」结构（与 `TeleportRequestManager`
  同构，独立一份以免 tp 请求与组队邀请相互干扰）。
- `PartyService` 用服务端组 API 直接改组：
  - `GroupManager.generateUniqueGroupID()` + `addGroup()` 在邀请者无队时新建队伍
    （邀请者 `EPlayerGroupRank.ADMIN`，新成员 `MEMBER`）。
  - `PlayerQuests.ServerAssignToGroup(groupId, rank, bypassMemberLimit:true)` 入队，
    `leaveGroup(force:true)` 离队/踢人。
  - 始终绕过原版人数上限，改用插件自己的 `party.maxMembers`（0=不额外限制，
    读 `GroupManager.getGroupInfo(id).members` 判定）。
- 因为用的是原版同一套服务端方法，组好的队伍**与原版组队互通、客户端 UI 正常显示**，
  并直接让 `/tp` 走「同队免确认直传」分支（`isMemberOfSameGroupAs` 对无队玩家返回
  false，单人不会被误判同队）。
- 配置：新增 `party` 段（`inviteExpirationSeconds`、`maxMembers`），在 WebPanel 的
  「rules」设置组里暴露可编辑（遵守 development-standards §8）。
- 注册：`PartyInviteManager` / `PartyService` 加入 `EssentialsContainerConfigurator`
  （插件作用域单例）。
- 文档：更新插件 `README.md` 与 `docs/well404.Essentials.md`（命令表、配置、组队小节）。

## 为什么

- 用户反馈「游戏内组队十分恶心」，希望插件能直接让玩家间进入同一队伍；并确认采用
  邀请+接受模式。组队同时解决了 `/tp` 跨队要确认的痛点（组队后队友互传免确认）。

## 影响的文件 / 范围

- 新增：`src/plugins/well404.Essentials/Party/{PartyInviteManager,PartyService}.cs`、
  `Commands/Party/CommandParty*.cs`（6 个命令）。
- 修改：`EssentialsSettings.cs`、`EssentialsYaml.cs`、`config.yaml`、`translations.yaml`、
  `EssentialsContainerConfigurator.cs`、`EssentialsWebPanelModule.cs`、插件 README、
  `docs/well404.Essentials.md`。

## 注意事项 / 后续

- 组 API（`ServerAssignToGroup` 等签名取自数据挖掘版并按本机 3.23.12.3 程序集符号核对，
  编译通过）**需真实服务器手测**：邀请→接受是否真正同队、客户端 UI 是否显示、踢人/离队
  是否生效、`maxMembers` 上限是否拦截。
- 组队逻辑触碰 `SDG.Unturned` 类型，无法纯逻辑单测；靠编译 + 服务器手测（纯逻辑测试
  仍 33 条全绿）。
- `/party kick` 仅队长（ADMIN）可用；离队用 `leaveGroup(force:true)`。
