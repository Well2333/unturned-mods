# 仓库代码审核后修复:线程安全 / 存入顺序 / 权限对等 / 健壮性

对 Vault 做了一次全面审核(两路独立 agent + 人工复核),并修复发现的问题。

## 修了什么

- **并发(高)**:`m_Data`(Players/Overrides)在 web/命令线程被读、同时主线程在写,`SaveAsync` 直接序列化活图。
  现在所有内存读写都过 `m_Lock`;`SaveAsync` 先在锁内**克隆快照**,再经 `SemaphoreSlim` 串行写盘。避免渲染与
  `/vault store` 并发时抛 `Collection was modified` 或 `vault.data.yaml` 撕裂。
- **存入可能脱离主线程(高)**:原先 `await GetMaxSlotsAsync(...).ConfigureAwait(false)` **在背包移除之前**,
  配了权限档位且权限后端异步时,后续 `getIndex/removeItem/Assets.find` 会在线程池续体上跑。改为**先算容量、再
  切主线程**,其后无 await,SDG 操作全程在主线程。(空档位时原本同步完成,所以之前测着正常。)
- **存入顺序(高→消除复制)**:改为**先确认 `removeItem` 成功(并复核 jar)再写入仓库**,杜绝陈旧/移动的格子导致
  物品被复制进仓库。targets 改存 `(page,x,y)` 而非 jar 引用。
- **权限对等(中)**:网页仓库的存/取现在强制 `commands.vault.store`/`.take`(与命令一致):无权限时按钮不显示、
  `InvokeAsync` 拒绝;**内容仍可只读查看**。`VaultPlayerMenu` 注入 `IPermissionChecker`。
- **健壮性(中/低)**:断线时 `user.Player?.Player` 判空;`GiveBack` 的 `FromBase64String` 包 try/catch(损坏状态
  不再中断整次取出,且会清掉坏条目防死循环);`config.MaxSlots` 读时 `Max(1,..)` 钳制;变体键 `Split('|',4)` 保整。
- **清理**:删除已无用的 `StoredItem.Uid`(改用变体匹配后只写不读)与 7 个死 i18n 键;管理面板 `ParseTiers` 报错改为
  **无参常量**并加中文(原为字符串插值,无法本地化)。
