# WebPanel:cloudflared 多镜像/重试/代理下载 + 隧道后台不阻塞启动 + 开服后醒目提醒

## 做了什么(well404.WebPanel 1.1.0 → 1.2.0)
- **多镜像 + 重试 + 代理下载**:`CloudflaredDownloader` 改为按 `web.tunnel.downloadMirrors` 列表
  **逐个尝试、先成功者胜**,每个源重试 `web.tunnel.downloadAttempts` 次(默认 2);自动遵循系统代理
  及 `HTTPS_PROXY`/`HTTP_PROXY`/`ALL_PROXY` 环境变量。条目支持含 `{asset}` 的完整 URL 模板、纯代理
  前缀(如 `https://ghproxy.com/`)、或空串(直连)。留空用内置默认(若干 GitHub release 代理在前、
  直连在后)。失败/无可用二进制时 `EnsureAsync` 返回 null,隧道跳过启动。
- **隧道后台化、不阻塞启动**:`OnLoadAsync` 不再 `await` 隧道启动;HTTP 监听起来后立即打印本地管理面
  地址,隧道(含 cloudflared 下载)放到后台 `BringUpTunnelAsync` 跑,就绪后单独打印公网管理面地址,
  随后按 `autoRestart` 进入保活监控。
- **开服后醒目提醒**:新增 `WarnAfterServerHostedAsync`——轮询 `Level.isLoaded && Provider.serverID`
  等服务器开服完成,再等隧道首次结果,然后若启动期间记录了任何问题(HTTP 监听失败、隧道未起来),用
  一条 `ERR` 横幅日志重新汇总提醒管理员,避免被开服刷屏淹没。

## 为什么
- 真机日志显示 cloudflared 直连 github.com 超时(连接 21s 失败),需要镜像/代理与重试才能在被墙网络下
  成功下载(用户反馈)。
- 带重试与多镜像后下载可能较久,若仍在加载路径上 `await` 会拖慢/卡住整个开服;后台化可避免。
- 启动告警淹没在大量开服日志里管理员看不到,用户要求开服(servercode 弹出)后再次醒目提示。

## 影响的文件 / 范围
