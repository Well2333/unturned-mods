# WebPanel:修复残留 cloudflared 进程跨重启占用面板端口(Windows)+ /menu 提示优化

## 做了什么(well404.WebPanel 1.2.2 → 1.2.3)
- **清理残留 cloudflared**:在绑定监听端口前(`OnLoadAsync`)以及卸载时(`OnUnloadAsync`),都会
  `KillOwnTunnelProcesses` 杀掉**本插件启动的** cloudflared(按可执行文件路径匹配:自动下载的便携
  二进制,或 `command` 配的绝对路径)。绑定失败时还会再清理一次并重试一次(共 2 次)。
- **/menu 提示优化**:隧道服务器刚开服、隧道还没起来时,`/menu` 短时拿不到链接;提示语改为「可能在
  启动中,稍候重试」,不再误导为「未配置 publicBaseUrl」。

## 为什么
- 真机(Windows)报 `failed to start HTTP listener ... address already in use`,第二次起服必现。
  根因:**Mono 在 Windows 上以 `bInheritHandles=TRUE` 启动带重定向 stdout/stderr 的子进程**,导致
  cloudflared **继承了面板的监听套接字**。若 cloudflared 在服务器崩溃/被强杀/非优雅退出后成为孤儿,
  它会一直占住面板端口,使下次 `HttpListener` 绑定失败、面板起不来。实测确认:**Linux/Mono 给套接字
  设了 CLOEXEC、不继承**,所以本地测试没复现——这是 Windows 专属问题。
- 启动前主动杀掉自己的残留 cloudflared 即可释放端口,让重启自愈;卸载时清理可避免优雅停服后残留。
- 非阻塞启动下隧道在后台建立,期间 `/menu` 无可用公网地址;旧提示让人误以为配置错误。

## 影响的文件 / 范围
- `well404.WebPanel/WebPanelPlugin.cs`(`KillOwnTunnelProcesses`、启动前清理 + 绑定重试、卸载清理、
  `m_TunnelCommand` 字段;新增 `using System.Diagnostics` / `System.Runtime.InteropServices`)
- `well404.WebPanel/translations.yaml`(`menu:unavailable` 文案)、`docs/well404.WebPanel.md`、`.csproj`(1.2.3)

## 注意事项 / 后续
- 进程匹配按**可执行文件全路径**(便携路径或 command 的绝对路径),避免误杀系统级、别人在用的 cloudflared;
  `command` 为 PATH 上的裸名时不做按名误杀(此时常见的自动下载便携路径仍会被清理)。
- 实测验证:优雅停服后无 cloudflared 残留、端口释放;模拟孤儿 cloudflared 后再启动,日志出现
  「terminated 1 leftover cloudflared ...」并成功绑定。整解 `-c Release` 0 警告 0 错误。
- 仅 WebPanel 改动,本轮只发布 well404.WebPanel 1.2.3。
- 关联:本问题是隧道(1.2.x 起才真正跑通)+ [[no-autonomous-push-or-release]] 流程下真机首次暴露。
